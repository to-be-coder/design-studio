# design-studio-reframe

The highest-leverage and least automatable move. A brief usually describes a local fix; your value
is spotting when it's a symptom of something larger — without scope-creeping into vagueness. This
skill creates the conditions for that insight. It does **not** supply it.

**Read `../design-studio-shared/CONVENTIONS.md` first.** Autonomy: 🔴 (run the 🔴 ritual).

## When to use
After `verify`. Runs standalone.

## Preconditions
- Expects the problem + research + verified assumptions. If thin, warn; you can still run the test
  on what's known.

## Process

1. **Lay out the evidence** research surfaced — especially any "these N things are the same
   primitive" pattern, and the recent change that might make a hard thing easy (new standard, tool,
   platform capability).
2. **Apply the reframe test** out loud, as structure for the user:
   - Is this the small or the big version of the problem?
   - A good reframe (a) stays grounded in the original ask, (b) explains it as a special case of a
     bigger pattern, (c) changes what "done well" looks like.
3. **Interrogate — do not assert** (🔴 ritual). Present the candidate framing(s) and ask the user
   which is true. Do NOT write the reframe yourself. Wait for their words.
4. **Gate — "no reframe needed" is a valid outcome.** If the brief was already the right frame, say
   so and record that. Never manufacture a pivot to look clever; reframe-for-its-own-sake is a
   junior tell.
5. **Record the decision** the user commits to — the reframe (with the framings rejected) or "brief
   is sound, proceed." Ladder it to the guiding principle.
6. **Update `00 Dashboard.md`** (stage = reframe, next = align via compile-spec, then scope-and-sequence).

## Handoff
If a reframe happened, suggest `design-studio-compile-spec` in **align** mode to socialize it with
stakeholders before deep work. Then `design-studio-scope-and-sequence`.
